多重待ちの待ち解除について

nn::os::WaitAny() 系の多重待ち OS-API では、多重待ち対象のいずれかの 同期オブジェクトがシグナル化されることで待ちが解除されますが、 待ちが解除されても、対象同期オブジェクトの状態は変更されません。

例えば、 nn::os::EventType オブジェクトと nn::os::SemaphoreType オブジェクトを WaitAny() している状態で、 nn::os::EventType オブジェクトが シグナル化された場合、 nn::os::WaitAny() の多重待ちは解除されますが、 nn::os::EventType オブジェクトは、その nn::os::EventClearMode に関わらず シグナル状態 が維持されます。

同様に、 nn::os::SemaphoreType オブジェクトがシグナル化された （他スレッドで ReleaseSemaphore() が呼び出された）場合でも、 WaitAny() の多重待ちは解除されますが、 nn::os::SemaphoreType オブジェクトのセマフォを獲得したりはしません。

そのため、 WaitAny() に成功したスレッドは、その返値で得た nn::os::MultiWaitHolderType* オブジェクトに関連付けられている nn::os::EventType オブジェクトや nn::os::SemaphoreType オブジェクト に対して、 TryWaitEvent() や TryAcquireSemaphore() などを発行して、 対象同期オブジェクトに対する要求処理を確実に行なうようにして下さい。

TryWait 系の API を使う理由は、上記の WaitAny() から TryWaitEvent() の間に 他のスレッドによって イベント を非シグナル状態にクリアされてしまう場合が あるためです。 TryAcquireSemaphore() についても同様で、 他のスレッドによって先にセマフォを獲得されてしまう可能性があります。

ここでは WaitAny() について説明しましたが、 TryWaitAny(), TimedWaitAny() についても同様の対応が必要です。
