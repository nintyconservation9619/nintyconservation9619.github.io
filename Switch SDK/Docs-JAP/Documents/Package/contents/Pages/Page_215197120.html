 nn::g3d ニュース

バイナリーリソースファイルのフォーマット更新

今後バイナリーリソースファイルの互換性を可能な限り維持するため、 bfres ファイルおよび bfsha ファイルのバイナリーフォーマットの調整を行いました。 これにより、以前のバージョンの 3D バイナリーコンバーターおよび 3D シェーダーコンバーターで作成したバイナリーファイルとの互換が切れますので、リソースの再コンバートが必要です。

プリミティブ最適化のデフォルトモードを Forsyth に変更

中間ファイルオプティマイザー のプリミティブ最適化の --mode オプションのデフォルト値を Normal から Forsyth に変更しました。これまで --mode を省略していた場合は最適化の結果に影響があります。また、Normal、Force、BruteForce についても処理頂点数の評価方法を Forsyth と同じ評価方法に変更しました。Normal、Force、BruteForce を指定していた場合にも最適化の結果に影響があります。処理頂点数の評価方法の詳細は中間ファイルオプティマイザのプリミティブ最適化の項目を参照してください。

この変更に伴い Maya プラグインのデフォルト値についても Forsyth に変更されています。

 nn::ui2d ニュース

同じシェーダーおよび同じバリエーションが連続した際にシェーダー設定をスキップする改善

同じシェーダーおよび同じシェーダーバリエーションが連続して描画される際に、シェーダー設定をスキップすることで CPU 負荷および GPU 負荷を軽減させました。

このため、同じブレンドのピクチャペインが連続するようにペインツリーを並び替えて頂くことで、さらに高速化させることができます。

特定プラットフォームでブレンドが同じピクチャペインを 150 個含むレイアウトデータを描画したところ、これまで 1 フレームあたり 0.20 ms かかっていた GPU 負荷が 0.12ms に改善されました。

gfx の GetGpuAddress の結果をキャッシュする改善

gfx の GetGpuAddress は高頻度で呼び出される関数のため、この結果をキャッシュすることで CPU 負荷を軽減させました。

特定プラットフォームで 320 個のペインを含むレイアウトデータを描画したところ、これまで 1 フレームあたり 1.28 ms かかっていた CPU 負荷が 1.25 ms に改善されました。

デフォルトコンストラクタによりプログラム上で生成したペインが表示されない不具合の修正

デフォルトコンストラクタを用いてプログラム上で生成した、テキストボックスペイン、ピクチャペイン、ウインドウペインが、画面上に表示されないことがある不具合を修正しました。

 nn::font ニュース

RectDrawer のワークメモリがオーバーランすることがある不具合の修正

nn::font::RectDrawer::Initialize() にて、ワークメモリサイズの計算が間違っていて、正しいアラインメント制約を満たしていないメモリを渡すと、オーバーランすることがある不具合がありこれを修正しました。

新しいバージョンでは、ワークバッファにアラインメント制約を要求するように仕様変更をし、誤った、アラインメントのメモリが設定された場合にアサートに失敗します。必要なアラインメント値は、nn::font::RectDrawer::GetWorkBufferAlignment() で取得するようにしてください。

不具合は、1524byte 以上アラインメント調整が行われる場合におこり、メモリ破壊や、意図しないメモリ属性の上書きによる、メモリ属性違反エラーが発生していました。以前(NintendoSDK v0.16よりも前)から、存在していた不具合なので、実際に問題が顕在化する状況は少なかったと予想しています。

nn::atk ニュース

nn::atk::StreamDataInfo の名前の変更

nn::atk::WaveSoundDataInfo との整合性を持つために nn::atk::StreamDataInfo 構造体の名前を変更しました。この変更により以下の機能が非推奨となります。

  • nn::atk::StreamDataInfo 構造体
  • nn::atk::StreamSoundHandle::ReadStreamDataInfo 関数

代わりに以下の同じ機能を提供する構造体と関数を追加したため、置き換えをお願い致します。

  • nn::atk::StreamSoundDataInfo 構造体
  • nn::atk::StreamSoundHandle::ReadStreamSoundDataInfo 関数

プリペア完了前のサウンドのリソース情報を取得する関数の追加

ウェーブサウンドとストリームサウンドに関して、リソース情報を取得する以下の関数を追加しました。

  • nn::atk::SoundArchivePlayer::ReadWaveSoundDataInfo
  • nn::atk::SoundArchivePlayer::ReadStreamSoundDataInfo
  • nn::atk::SoundArchivePlayer::ReadStreamSoundRegionDataInfo

nn::atk::SoundArchivePlayer::ReadWaveSoundDataInfo は既存の関数 nn::atk::WaveSoundHandle::ReadWaveSoundDataInfo とは異なり、ロードが完了していればプリペア完了前でも情報を取得することが可能です。

nn::atk::SoundArchivePlayer::ReadStreamSoundDataInfo は既存の関数 nn::atk::StreamSoundHandle::ReadStreamSoundDataInfo とは異なり、サウンドを再生していなくても情報を取得することができます。ただし、ストリームサウンドの再生中は情報の取得に失敗するためご注意ください。

nn::atk::SoundArchivePlayer::ReadStreamSoundRegionDataInfo からはリージョン情報を取得することができます。具体的には、ストリームサウンドに設定されたリージョン名から、開始サンプル位置、終了サンプル位置、リージョン番号を取得することができます。ただし、ストリームサウンドの再生中は情報の取得に失敗するためご注意ください。

それぞれの機能の詳細は API リファレンスを参照ください。

ハードウェアアクセラレータを利用した Opus のストリーム再生の追加

ハードウェアアクセラレータを利用した Opus ストリームサウンド再生に対応しました。

ハードウェアアクセラレータである HardwareOpusDecoder の初期化をしたのち、再生時に nn::atk::SoundStartable::StartInfo によりデコード方式を変更することでこちらの機能を利用することができます。

詳細は、プログラミングガイド／ストリームサウンドの「Opus 再生」の項目や、 AtkStreamSound サンプル、以下の API リファレンスを参照してください。

  • nn::atk::InitializeHardwareOpusDecoder
  • nn::atk::SoundStartable::StartInfo::streamSoundMetaInfo.decodeMode
    • nn::atk::SoundArchive::StreamSoundInfo::decodeMode
  • nn::atk::FinalizeHardwareOpusDecoder

プログラミングガイドの改訂

  • ストリームサウンドのページの構成を見直しました。
  • ストリームサウンドのページに、ストリームサウンドを再生する際に使用されるストリームバッファに関する説明を追加しました。
