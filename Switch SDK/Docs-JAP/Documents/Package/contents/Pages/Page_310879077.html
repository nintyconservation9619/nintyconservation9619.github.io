概要

開発支援機能の設定を変更できます。

注意：

一部の項目はPROD MODEでは表示されません。DEV MODEでのみ設定可能です。

使い方

Initial Boot Menu

開発機で最初に起動されるメニューを設定します。デフォルト値は ｢DevMenu｣ です。Homeメニューがインストールされていない場合は ｢DevMenu｣  しか設定できません。

  • 設定値がDevMenuのとき、本体起動時にDevMenuが立ち上がります。
  • 設定値がHomeMenuのとき、本体起動時にHOMEメニューが立ち上がります。

変更後の設定を有効にするには、開発機の再起動が必要です。

HOMEメニューが起動される設定からDevMenuが起動される設定への切り替えには、 DevMenuCommand のdebug switch-menuコマンド等を利用します。

Memory Mode

開発機のメモリサイズをエミュレートします。実際に搭載されているメモリサイズよりも大きなサイズ設定は無視されます。
設定可能なメモリサイズは以下になります。

  • Auto (デフォルト値）
  • 4 GB
  • 6 GB

変更後の設定を有効にするには、開発機の再起動が必要です。

注意：
  • SDEVでのみ有効な設定です。EDEVは設定値に関係なく常にメモリサイズ4 GBで動作します。

メモリ情報の修正が必要なSDEVの場合（NX Addon 4.4.0以降）

Nintendo Developer Portalの記事「2017年 2月～3月に製造・販売された SDEV では メモリが 4GB までしか使えない」の対象になるSDEVでは、メモリ情報の修正が行われていない場合、以下のように赤字で「(Needs Memory Repair)」と表示されます。

これが表示されているSDEVはメモリ情報の修正が必要ですので、Nintendo Developer Portalの記事で紹介しているとおりツールを適用して修正してください。

なお、Memory Modeが「4 GB」に設定されているときはこの情報は表示されません。メモリ情報の修正が必要なSDEVかどうかを判断するためには、Memory Modeを「Auto」か「6 GB」に設定してください。

Fill Memory with Non-Zero Value

開発機においては、確保したメモリ領域を非ゼロ値で埋めるか否かを設定できます。デフォルト値は ｢Enabled｣ です。
製品機では確保したメモリ領域の初期値は不定です。必要に応じて初期化して利用してください。
変更後の設定を有効にするには、開発機の再起動が必要です。

User Exception Handler

アプリケーションのユーザー例外ハンドラを有効にするか無効にするかを設定できます。デフォルト値は ｢Enabled｣ です。
変更後の設定を有効にするには、開発機の再起動が必要です。

Red Screen at System Error

システムエラー時にエラー画面（赤画面）に飛ばすか否かを設定できます。デフォルト値は ｢Disabled｣ です。
変更後の設定を有効にするには、開発機の再起動が必要です。

Initial Tick Value

チックの初期値を設定できます。デフォルト値は「0」です。「0」の他、「7日」「14日」「30日」「90日」「カスタム」から選択できます。
「カスタム」を選択した場合、「0x」から始まる16進数の値を入力することで任意のチック値を初期値とすることができます。
変更後の設定を有効にするには、開発機の再起動が必要です。

FS Access Log Mode

アプリケーションのFSアクセスのログ出力を設定できます。デフォルト値は「OFF」です。
アプリケーションを起動する前に設定してください。開発機を再起動すると設定はデフォルト値に戻ります。
詳しい使い方は FSライブラリ - デバッグ機能 - アクセス頻度確認  のページを参照してください。

FS Speed Emulation Mode

開発機にインストールされたアプリケーションのFSアクセス速度のエミュレーション機能を設定できます。デフォルト値は「OFF」です。
アプリケーションを起動する前に設定してください。開発機を再起動すると設定はデフォルト値に戻ります。
詳しい使い方は FSライブラリ - デバッグ機能 - アクセス速度依存確認  のページを参照してください。

Fill NAND Free Space

NANDの空き容量を埋めることができます。「Fill」 で空き容量を埋め、「Empty」で埋めた容量を解放します。

Mii Testing Mode

似顔絵ライブラリに関する開発支援機能の有効・無効を切り替えることができます。デフォルト値は ｢Disabled｣ です。

Import Database

MiiデータベースをmicroSDカードからインポートします。

microSDカードにmiiディレクトリを作成しデータベースファイル(拡張子:dat)を入れてください。

この機能を利用するには、Mii Testing Modeを「Enabled」に変更する必要があります。

注意：

開発機以外でエクスポートしたファイルは利用できません。

Export Database

MiiデータベースをmicroSDカードにエクスポートします。

エクスポートしたデータベースファイルは、microSDカードのmiiディレクトリ以下に日付名(yyyyMMddHHmmss.data)で出力します。

この機能を利用するには、Mii Testing Modeを「Enabled」に変更する必要があります。

Format Database

Miiデータベースをフォーマットします。

この機能を利用するには、Mii Testing Modeを「Enabled」に変更する必要があります。

Export CharInfo

Miiデータベースをnn::mii::CharInfo形式でエクスポートします。

エクスポートを実行するとmicroSDカードのmiiディレクトリ以下に日付名(yyyyMMddHHmmss)でディレクトリが作成されます。
作成されたディレクトリの下に各nn::mii::CharInfo(拡張子:data)と"nicknames.txt"が出力されます。
"nicknames.txt"ファイルには、各nn::mii::CharInfoのニックネームが記載されています。

この機能を利用するには、Mii Testing Modeを「Enabled」に変更する必要があります。

Exhibition Mode

exhibition modeで動作するかを切替えることできます。デフォルト値は「Disabled」です。

変更後の設定を有効にするには、開発機の再起動が必要です。

Background Download Stress Test

バックグラウンドでダウンロードタスクが実行され続けるようになります。デフォルト値は「Disabled」です。
本機能を利用するためにはネットワークに接続できる必要があります。
本機能は本体保存メモリーを摩耗させますので、必要なときだけ「Enabled」に設定してください。

変更後の設定を有効にするには、開発機の再起動が必要です。

Crash Dump

CPU Crash Automatic Dump

チェックを入れた場合、アプリケーションがNN_ABORTなどでクラッシュした際、CPUのクラッシュダンプファイルを自動で出力するようになります。
ダンプ機能の詳細については ダンプ機能の使用方法 を参照してください。
変更後の設定を有効にするには、開発機の再起動が必要です。 

CPU Crash Full Dump

チェックを入れた場合、CPUのクラッシュダンプファイルにユーザメモリの状態も含まれるようになります。
チェックを入れなかった場合と比較してダンプにかかる時間が長くなるため、必要に応じてチェックを入れてください。 
変更後の設定を有効にするには、開発機の再起動が必要です。 

GPU Crash Automatic Dump

チェックを入れた場合、アプリケーションがGPUエラーを発生させた際、GPUのクラッシュダンプファイルを自動で出力するようになります。
ダンプ機能の詳細については GPU クラッシュダンプ機能 を参照してください。 
変更後の設定を有効にするには、開発機の再起動が必要です。 

Terminate Application on GPU Error

GPUエラーを発生させたアプリをシステムによって強制的に終了させる設定を切り替えることができます。デフォルト値は「Enabled」です。
変更後の設定を有効にするには、開発機の再起動が必要です。

HTC Disconnection Emulation Mode

ホストPCと通信を行う機能の切断状態をエミュレーションする機能を有効にするか無効にするかを設定できます。デフォルト値は「Disabled」です。
変更後の設定を有効にするには、開発機の再起動が必要です。
「Enabled」状態では、以下のような挙動をするようになります。

  • Target Managerと開発機が接続した状態でもHTC、HTCSライブラリ、および、ホストPC上のファイルシステム操作の機能が切断時と同じ挙動をするようになります
  • 一部機能を除いてホストPCからの開発機の制御が行えなくなります
    • これらの操作を行いたい場合は「Disabled」に設定する必要があります
  • デバッグ及びログ出力は可能になっています
    • デバッグについてはホストPCからのアプリケーション実行は行えなくなりますので、あらかじめアプリケーションをインストールした上で「ビルド済プログラムをデバッグする」の方法を使用してください

SD Card Logging Mode

ログをmicroSDカードへ出力する機能を有効にするか無効にするかを設定できます。デフォルト値は「Disabled」です。
変更後の設定を有効にするには、開発機の再起動が必要です。

保存

「Enabled」状態では、以下の流れでmicroSDカードにログを出力することができます。

  1. microSD カードを挿した状態で本体を起動します
  2. microSDカードのルートディレクトリのファイル"NxBinLogs/XXXX.nxbinlog"にログが出力されます
  3. ログの保存が必要な期間が終わったらmicroSDカードを抜いてください。
参考：

XXXXは0000から始まり、同名のファイルがある場合はインクリメントされていきます。
インクリメントは9999でストップし、それ以上のログは出力されません。

閲覧

  1. Log Viewerを起動します。
  2. Menuから"Open NX binary log"をクリックします。
  3. 保存した"nxbinlog"ファイルを選択します。

以下のようなタブが追加されます。フィルタリングなどの機能はTarget Managerを受信するタブを同等です。
 

(※) Log Viewerの詳しい使い方についてはLog Viewerのドキュメントを参照してください。

No Vsync Capability

開発機においてグラフィックスAPIによるVsyncオフを許可するかどうかを設定します。デフォルトでは「Disabled」にされており、グラフィックスAPIによりVsyncをオフにする事はできません。また、製品実機ではグラフィックスAPIによるVsyncオフは許可されません。詳細は nvnWindowSetPresentInterval() の説明を参照してください。

Gpu Timeout

参考：

NX Addon 4.0.0以降で使用可能です。

GPUタイムアウト設定をします。デフォルト値は「Enabled」です。
GPUタイムアウトの詳細については グラフィックス (NX Addon) の「1回のドローコールにかけることのできる時間の制限」を参照してください。
変更後の設定を有効にするには、開発機の再起動が必要です。

Graphics Firmware Memory Margin

参考：

NX Addon 4.0.0以降で使用可能です。

グラフィックスファームウェアメモリマージンの設定を切り替えます。デフォルト値は「Enabled」です。
グラフィックスファームウェアメモリマージンの詳細は グラフィックス (NX Addon) の「グラフィックスファームウェアが使用するメモリ」を参照してください。
変更後の設定を有効にするには、開発機の再起動が必要です。

Ease Nro Restriction

nrrファイルは必ずROM-FSに格納し、 ROM-FSのルートディレクトリを"/"としたときに"/.nrr"ディレクトリ直下に 格納する必要があり、また、VSIのRAW形式では使用できない制限があります。
この制限を取り払い、VSIのRAW形式での使用や、ホストPC上のnrr, nroファイルを使用することが可能になるように、設定を切り替えることができます。
デフォルト値は「Enabled」で、制限が取り払われている状態です。

Operation Mode Policy

開発用に、動作モードの制御ポリシーを変更することができます。

  • Operation Mode Policyが「Auto」の場合は、ドックの着脱に応じて動作モードが変化します。
  • Operation Mode Policyが「Handheld」の場合は、動作モードが携帯機モードに固定されます。
  • Operation Mode Policyが「Console」の場合は、動作モードが据え置きモードに固定されます。

デフォルト値は「Auto」です。
変更後の設定を有効にするには、開発機の再起動が必要です。

Performance Mode Policy

開発用に、性能モードの制御ポリシーを変更することができます。

  • Performance Mode Policyが「Auto」の場合は、ドックの着脱に応じて性能モードが変化します。
  • Performance Mode Policyが「Normal」の場合は、性能モードがノーマルモードに固定されます。
  • Performance Mode Policyが「Boost」の場合は、性能モードがブーストモードに固定されます。

デフォルト値は「Auto」です。

注意：
  • Performance Mode Policy「Boost」はSDEVでのみ使用可能です。 EDEVでは使用できません。
  • Performance Mode Policyを「Boost」にすると、ボード全体の消費電力の上限の都合上、本体側からの電源供給でUSBデバイスを動作させることはできなくなります。

NX Addon 5.2.0より前の環境では、変更後の設定を有効にするために開発機の再起動が必要です。

CPU Overclock

CPUのクロックレートを、製品で使用可能なクロックレートよりも高いクロックレートに固定できます。

  • CPU Overclockが「Disabled」の場合、CPUクロックレートは製品で使用可能な1020 MHzになります。
  • CPU Overclockが「Enabled」の場合、CPUクロックレートは1785 MHzになります。デバッグビルド環境や開発時固有の機能による処理負荷増を吸収する用途を想定しています。

デフォルト値は「Disabled」です。

注意：
  • SDEVのみで使用可能です。EDEVでは使用できません。
  • EDEVでCPU Overclockを「Enabled」にしても効果はありません。「Disabled」時と同じCPUクロックレートで動作します。
  • CPU Overclockが「Enabled」の時は「Disabled」の時に比してSDEVの表面温度が上昇します。取り扱いにはご注意ください。
  • CPU Overclockが「Enabled」の 時はPCMライブラリによる消費電力測定結果およびCPUプロファイラの結果は、無効時の結果と大きく異なる値になります。これらの値を製品環境の参考にする場合はCPU Overclockを「Disabled」にしてください。

NX Addon 5.2.0より前の環境では、変更後の設定を有効にするために開発機の再起動が必要です。

Reset Pseudo Device ID

疑似本体IDをリセットします。
リセット後は各種アプリケーションで取得する疑似本体IDの値はリセット前のものと異なるものを返すようになります。

Thermal Burn Prevention Mode

低温やけど回避機能の有効・無効を切り替えます。

低温やけど回避機能が「Enabled」の場合、以下の全てを満たす状況が長時間継続したときに強制的にスリープします。

  • 携帯モード
  • ボタン入力等の明らかなユーザ操作がない
  • 推定表面温度が一定以上

開発機でのデフォルト値は「Disabled」ですが、製品機に近いファームウェアへの初期化をした場合、デフォルト値は「Enabled」です。

Idle Time Detection や自動スリープの発動時間などの他の本体設定は、本機能の挙動には影響しません。
