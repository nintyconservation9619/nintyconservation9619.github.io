全般

SD カードを挿した状態で起動途中に電源断、もしくは再起動を繰り返し行うとブリックすることがある

症状
  • SD カードを挿入している状態で電源断もし再起動を繰り返すと、 画面が暗い状態のまま処理が進まなくなることがあります。
  • この状態になると再起動しても繰り返し同じところで止まるようになります。
回避策

SD カードを抜いて再起動をしてください。

 

再起動後、 DevMenuCommand で SD カードを初期化するか、 PC で SD カードをフォーマットしてください。

正常な SD カードであれば挿した状態でも起動することが出来ます。

 

修正予定

次回以降のリリースで修正予定です。

Audio

アプリケーションを VSI から実行していると、徐々に再生ボリュームが小さくなることがある

症状
  • オーディオ出力、オーディオレンダラを利用しているアプリケーションを VSI から実行していると、アプリケーションから再生される音のボリュームが徐々に小さくなっていく場合があります。
  • スピーカー出力、TV 出力、ヘッドフォン出力に関わらず、ボリュームは下がったままとなります。
  • ただし、nn::audio::CircularBufferSink で取得される波形データはこの影響を受けず、正しいボリュームで出力され続けます。
回避策

この問題は VSI 経由でアプリケーションを起動した場合にのみ発生し、アプリケーションを本体にインストールした状態、もしくは RunOnTarget によってアプリケーションを起動した場合には発生しません。

また VSI 実行中に発生した場合も、以下の操作を行うことでボリュームが正しい設定に回復します。

  • Home ボタンを押し、システムメニューもしくは DevMenu に一度遷移させた後、再度 Home ボタンを押し、アプリケーションに戻る。
修正予定

次回以降のリリースで修正予定です。

複数個のオーディオレンダラを同時に使用した時に正しく DSP リソース配分が効かない事がある

症状

複数個のオーディオレンダラを同時に使用した時に、正しく DSP リソース配分が効かない事があります。
具体的には、アプリケーションが複数個のオーディオレンダラを同時に利用している時に、オーディオフレームあたりの DSP 処理時間の合計値が 3500 us を超えている場合に発生します。

回避策

複数個のオーディオレンダラを使用するときは、オーディオフレームあたりの DSP 処理時間の合計値が 3500 us 以下になるようにしてください。

修正予定

Joy-Con とインターネットをともに無線で接続している場合にシステムが不安定になることがある

症状

Joy-Con 無線接続時かつネットワーク無線接続が利用可能な状況で、以下のような症状が起こる場合があります。

  • NIFM ライブラリへの、ネットワーク利用要求の提出結果がいつまでも確定しない
  • スリープ状態に入らず、 Joy-Con との接続がいつまでも切れない
  • メニューの画面がいつまでも切り替わらない
  • メニューの「インターネット設定」でおこなう無線アクセスポイントのスキャンがいつまでも終了しない

また、製品相当の状態では、一定時間のはまりのあとに致命的なエラーが発生する場合があります。

回避策

Joy-Con を一度本体に接続することではまりから抜けることができます。

また、無線のインターネット接続設定を登録せず、常に有線接続しか利用されない状況にすることで問題の発生を回避できます。

症状

  • WiFiをフライトモードに入れる。

      • 設定マネージャツール（SettingsManager.nca）を使って以下の設定ファイルをインポートする。

        {
    "head": {
        "version": "1.0"    // フォーマットバージョン
    },
    "body": {
        "wifi_settings": {
            "wifi_enabled": {
                "current": false,
                "default": false
            }
        }
    }
}
  • 使用機材がSDEVの場合は、本体背面のDIPスイッチで、１，２番のスイッチ（左から１，２番目）を下側に入れる。SDEV MPであれば、「WiFi1/BT」「WiFi2」と刻印がされています。

 ディスプレイリリースフェンスを待つなどして VSync 直後にアプリケーションの GPU コマンドが実行されると、システムの GPU 時間割り当て開始と競合するため、GPU の実行タイミングが不安定になります。

回避策

VSync 直後に GPU コマンドが実行されないよう、VSync 後にコマンドを発行するようにしてください。システムに GPU 時間が割りあたっている間にアプリケーションのコマンドが発行されても問題ありません。アプリケーションに GPU 時間が割りあたった直後からコマンドが実行されます。

修正予定

将来のバージョンで修正予定です。

NVN

ディスプレイがフレームバッファを更新するタイミング以外のタイミングでもディスプレイリリースフェンスがシグナルする

症状

nvnWindowAcquireTexture() が返す NVNsync オブジェクト（ディスプレイリリースフェンス）は通常、ディスプレイコントローラーがスキャンするフレームバッファが VSync 時に切り替わるタイミングでシグナルしますが、次のタイミングでもシグナルする場合があります。

  • フォーカス状態がインフォーカス状態と他の状態の間で遷移するタイミング
  • ドックイン・ドックアウトや HDMI ケーブル挿抜によりディスプレイが切り替わるタイミング
  • スリープにする・スリープを解除するタイミング
  • 本体設定でテレビ出力の解像度を変更したタイミング

これらの場合、ディスプレイリリースフェンスがシグナルしても nvnQueuePresentTexture() で発行された GPU コマンドが完了したことが保証されません。アプリケーションの CPU 処理が GPU 処理の完了をディスプレイリリースフェンスによって判定していると、上記タイミングで GPU が使用中のコマンドバッファやその他のバッファを破壊、もしくは書き換えてしまう可能性があります。

回避策

GPU 処理の完了待ちは、ディスプレイリリースフェンスによって間接的に待つのではなく、nvnQueueFinish() もしくは nvnQueueFenceSync() と nvnSyncWait() によって直接待つようにしてください。

修正予定

修正可否を検討中です。修正時期は未定です。

ドックイン・ドックアウト時に PresentTexture(), AcquireTexture() の呼出しが数百 ms ブロックされることがある

症状

本体のドックイン・ドックアウトによってディスプレイが切り替わるタイミングで nvnQueuePresentTexture() や nvnWindowAcquireTexture() の呼出が 200 ms から 300 ms 程度の時間ブロックされることがあります。
これによってアプリケーションの描画ループが一定間隔で回らなくなり、アプリケーションの実装によっては音ずれなどの症状が表れる場合があります。

回避策

動画の再生などの長時間に渡って画面と音声の同期が必要なシーンでは描画ループが一定間隔で回ることを期待せずに、明示的に画面と音声の同期を取るように実装する必要があります。

修正予定 

修正可否を検討中です。修正時期は未定です。

GLSLC 関連

  • GLSL built-in uniforms gl_WarpSizeNV, gl_WarpsPerSMNV, and gl_SMCountNV do not work correctly.  Applications can work around using NVN device properties.
  • GLSL shaders using NV_fragment_shader_interlock features will consume more scratch memory than similar shaders that do not use the feature.  We are investigating whether this requirement can be reduced.
  • Multi-threading does not work with the GLSLC compiler.  If applications wish to use multiple compile objects in parallel, they will need to do so in parallel processes instead.  This bug is being worked on for the next release.
  • EOL markers must be present in #included files or else a compilation error will occur.
  • Compute shader data subsections inside a GPU code section do not start with the magic number GLSLC_GPU_CODE_SECTION_DATA_MAGIC_NUMBER.
  • The TX1 versions of the GLSLC library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms should be functionally correct though, and convergence of the two will happen in future changes.
  • FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension.  When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface and mediump/lowp types on the other side of the interface could produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages will produce a linker error.
    • ints/uints can not be implicitly converted to floats qualified with mediump/lowp qualifiers.  This can be worked around by explicit constructing float types from the int/uint variables (similar to explicit casting in C/C++).  This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp _as well as_ ES shaders regardless of whether the extension GL_EXT_shader_implicit_conversions is used.
    • FP16 math when used with imageStore may cause corruption in certain cases. This issue is being investigated and will be fixed in an upcoming release.

素早いドックイン・ドックアウトによりシステムがハングすることがある

症状

アプリケーションからディスプレイへ高い負荷がかかっているときに素早くドックイン・ドックアウトを繰り返すと、システムがハングすることがあります。

システムのクラッシュが原因によるハングではないため、クラッシュログはダンプされません。

発生した場合、 TargetManager から本体をリセットするか、電源ボタンを押し続けて本体の電源を切ってから起動し直してください。

回避策

3 秒未満の間隔でのドックイン・ドックアウトを繰り返さないようにしてください。
