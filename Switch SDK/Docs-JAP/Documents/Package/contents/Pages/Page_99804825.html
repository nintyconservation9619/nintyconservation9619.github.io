ディスクリプタスロットの仕様変更について

 

nn::g3d

NintendoSDK 0.8.0 では変更はありません。

 

nn::font / nn::ui2d / nn::vfx

ui2d のコンスタントバッファの扱いの変更

パフォーマンス、メモリ効率向上のためレイアウトライブラリが使用するコンスタントバッファを nn::ui2d::GraphicsResource へ集約しました。

この変更によりユーザーはアプリケーションが使用するコンスタントバッファのサイズを nn::ui2d::GraphicsResource の初期化時に明示的に渡す必要があります。

また、nn::ui2d::Layout::CalculateMtx 内でコンスタントバッファを更新する前に更新準備処理、更新後に更新終了処理を呼び出す必要があります。

詳しくは Ui2dSimple サンプルデモを参照してください。

参考：

現在、コンスタントバッファに関する処理の排他制御が行われていないため、複数スレッドから同時に nn::ui2d::Layout::CalculateMtx を呼び出すと正しくコンスタントバッファが更新されません。

複数スレッドから更新したい場合は nn::ui2d::GraphicsResource::Setup でコンスタントバッファをスレッド数分作成し、スレッドごとに操作するコンスタントバッファのインデックスを操作することでこの問題を回避することが可能です。

将来的にはコンスタントバッファの管理をユーザー側で行うように変更することで問題を解消する見込みです。

NintendoSDK 0.7.0 で行われた NintendoWare グラフィックスの変更点について説明します。

低レイヤグラフィックスライブラリの変更

NintendoWare グラフィックスモジュール( nn::g3d / nn::ui2d / nn::vfx ) ではこれまで、OpenGL を低レイヤグラフィックスライブラリとして利用してきましたが、NintendoSDK 0.7.0 にて nn::gfx ライブラリ に変更しました。

変更後は以下のような構成になります。

この変更により、各モジュールの旧nw空間から移行するにあたって、ユーザー側でコード書き換えが発生する仕様変更が行われています。

 

グラフィックスデバイスについて

nn::gfx ライブラリは、テクスチャやサンプラなどのグラフィックスオブジェクトの初期化に nn::gfx::Device オブジェクトを要求します。よって、nn::g3d / nn::ui2d / nn::vfx ランタイムでも、システムの初期化であったり、リソースの初期化メソッドにて nn::gfx::Device オブジェクトを要求するような仕様変更が施されています。

ディスクリプタプールについて

テ クスチャオブジェクトやサンプラオブジェクトは、ディスクリプタプールに登録しディスクリプタスロットを取得した上で利用されます。このディスクリプタ プールは、アプリで１つしか用意されないので、font / ui2d / vfx  ランタイム内部で生成されるオブジェクトはコールバック関数を通して、生成されたオブジェクトの情報をアプリ側に通知し、アプリ側でディスクリプタプール への登録、ディスクリプタスロットの取得を行っていただく必要があります。

ディ スクリプタプールを利用することで１つのオブジェクトのレンダリングに利用できるテクスチャやサンプラの数の上限が飛躍的に増加しますが、ディスクリプタ プールに登録可能なグラフィックスオブジェクト数には制限がありますし、用意可能なディスクリプタプールは、NintendoWare、アプリケーショ ン、OS で１つになります。

テクスチャオブジェクト 2^20個 ( 1048576 個 )
サンプラオブジェクト 2^12個 ( 4096 個 )

// テクスチャオブジェクトをディスクリプタプールへ登録するコールバック関数の記述例
bool AllocateSlotForTexture( nn::gfx::DescriptorSlot* dstSlot, const nn::gfx::TextureView& textureView )
{
    // 空いているディスクリプタインデックスを確保する
    int emptyTextureSlot = GetEmptyTextureDescriptorSlot();

    textureDescriptorPool.BeginUpdate();
    {
        textureDescriptorPool.SetTextureView( emptyTextureSlot, &textureView);
    }
    textureDescriptorPool.EndUpdate();

    // ディスクリプタインデックスからディスクリプタスロットを取得する
    textureDescriptorPool.GetDescriptorSlot( dstSlot, emptyTextureSlot );
    return true;
}

// サンプラオブジェクトをディスクリプタプールへ登録するコールバック関数の記述例
bool AllocateSlotForSampler(nn::gfx::DescriptorSlot* dstSlot, const nn::gfx::Sampler& sampler)
{
    // 空いているディスクリプタインデックスを確保する
　  int emptySamplerSlot = GetEmptySamplerDescriptorSlot();

    samplerDescriptorPool.BeginUpdate();
    {
        g_samplerDescriptorPool.SetSampler( emptySamplerSlot, &sampler );
    }
    samplerDescriptorPool.EndUpdate();

    // ディスクリプタインデックスからディスクリプタスロットを取得する
    samplerDescriptorPool.GetDescriptorSlot( dstSlot, emptySamplerSlot );
    return true;
}

// テクスチャ、サンプラ をコールバック経由でディスクリプタプールに登録する

// nn::font の例
nn::font::ResFontBase::RegisterTextureViewToDescriptorPool( AllocateSlotForTexture );
nn::font::RectDrawer::RegisterSamplerToDescriptorPool( AllocateSlotForSampler );

// nn::vfx の例
nn::vfx::System::RegisterTextureViewToDescriptorPool( AllocateSlotForTexture );
nn::vfx::System::RegisterSamplerToDescriptorPool( AllocateSlotForSampler );
nn::vfx::Resource::RegisterTextureViewToDescriptorPool( AllocateSlotForSampler );

テクスチャ リソース について

g3d /  vfx モジュールで利用するテクスチャファイルは、gfx テクスチャリソースを利用します。 gfxテクスチャリソースは、NW4Fテクスチャ形式と互換性があり、これまでに作成したテクスチャリソースはそのままご利用いただけます。ただし、 vfxモジュールでは、NitendoSDK 0.6.0 まではタイリングテクスチャを利用可能でしたが、NitendoSDK 0.7.0 以降でWindowsプラットフォームで利用する場合、リニアテクスチャにコンバートする必要があります。
ui2d ではこれまで通り、TGAテクスチャを利用可能です。

nn::gfx ライブラリを利用した実装について

現状、NitendoWare グラフィックス各モジュールの nn::gfx ライブラリ内機能 の対応状況は以下の通りです。

これらの対応状況は、今後サポートするプラットフォームに対して最適に動作するよう順次対応を行っていく予定です。

機能 利用状況
メモリプール 利用する
テクスチャ ディスクリプタプール 利用する
サンプラ ディスクリプタプール 利用する
コンスタントバッファ ディスクリプタプール 利用しない
パイプラインモード 利用しない
ルートシグネチャモード 利用しない

各モジュールの変更点

nn::g3d の変更点

レンダーステートについて

nw::g3d ではResRenderInfoクラスとResRenderStateクラスという似た機能を提供するクラスが存在していたため、クラスを整理し、 ResRenderStateクラスを削除、ResRenderInfoクラスにその役割を引き継ぎました。この変更により、nw::g3dで使用してい たモデルをnn::g3d向けにコンバートし直した場合、3dEditorで設定したレンダーステートの情報はそのままではモデルデータに含まれていませ ん。レンダーステートとして必要な情報は、シェーダに描画情報のアノテーション(renderinfo)を記述し、3dEditorで適切に設定し、 ResRenderInfoクラス経由で取得する必要があります。

nn::ui2d、nn::font の変更点

フォント リソース

フォントのフォーマットが変わりましたので、フォントバイナリを作り直す必要があります。

その他

スケーラブルフォントファイルを使用する前に nn::fontll::ScalableFontEngineHelper::Decode を呼び出す処理が必要になっています。

nn::ui2d, nn:font の組み込みシェーダの再変換スクリプトの追加

ユーザが nn::ui2d, nn::font の組み込みシェーダを再変換するためのスクリプトを追加しました。
組み込みシェーダにトラブルが発生した場合、このスクリプトを使って再変換することができます。

※ 変換済の組み込みシェーダが同梱されていますので、通常はユーザによってシェーダを再変換する必要はありません。

シェーダの変換スクリプトは以下の場所にあります：

  • Tools/Graphics/ShaderConvertScripts/BuildUi2dShaders.Win32.bat （ nn::ui2d 用）
  • Tools/Graphics/ShaderConvertScripts/BuildNnFontShaders.Win32.bat （ nn:font 用）

バッチファイルをダブルクリックすることでシェーダの再変換が行われます。
再変換した組み込みシェーダをランタイムライブラリに反映させるためには nn::ui2d, nn::font のランタイムライブラリの再ビルドが必要です。
