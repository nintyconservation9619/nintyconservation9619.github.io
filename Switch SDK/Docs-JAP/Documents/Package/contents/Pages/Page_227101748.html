概要

いくつかのサンプルケースについて、セーブデータの設計例を記載します。

ケース1: 各ユーザーが1人で遊ぶRPG

ネットワークを介して他人と一緒に遊ぶ仕様がない、各ユーザーが1人で遊ぶRPGです。

ユーザーがそれぞれにプレイヤーキャラクターを作成し、他のユーザーと関わりはなくそれぞれにストーリーを進めます。

セーブデータの構築例

各アカウントのアカウントセーブデータ領域に、以下のデータを保存します：

  • ストーリーの進捗状況
  • プレイヤーキャラクター
    • レベル、ステータス
  • アイテム、装備品

ケース2: 多人数同時プレイに対応したスポーツゲーム

ユーザーアカウントを所有する/しないにかかわらず、複数のユーザーが同時にプレイするアプリケーションです。

複数人同時プレイのスポーツ要素があり、ユーザーごとに成績も記録される一方で、アプリケーション内の要素の解放（アンロック）は本体全体で共有されます。
例えば、2人でテニスのダブルスをしたり、レースで競走したりするような場合で、その本体では誰がプレイしても同じステージやコースを遊べます。

セーブデータの構築例

各アカウントのアカウントセーブデータ領域に、以下のデータを保存します。

  • ユーザーごとの勝率
  • ユーザーごとのコースやステージの開放状況
  • ユーザーのアンロックした装備やキャラクター
  • ユーザーのよく使う装備の組み合わせや、コスチュームのコーディネート
  • コースレコード

起動時などに全アカウントのアカウントセーブデータ領域から以下の情報を集計し、アプリケーションに反映させます。

  • その本体のユーザー全体でのランキング
  • その本体のユーザー全体でのコースやステージの開放状況
  • その本体のユーザー全体でのアンロックした装備やキャラクター
