概要

AuthoringTool は、ビルドしたプログラムやリソースデータを元に、NX 実機にインストール及び実行可能な形式のファイルを作成するためのコマンドラインツールです。

配置場所

このツールは Nintendo SDK に含まれており、以下の場所に配置されています。

Tools/CommandLineTools/AuthoringTool/AuthoringTool.exe

またこのツールは上記ディレクトリの次の DLL に依存しています。実行する際は以下の DLL を exe と同じ場所に配置してください。

  • ApplicationControlPropertyModel.dll
  • AuthoringTool.Resources.dll
  • en\AuthoringTool.Resources.resources.dll
  • ContentArchiveLibrary.dll
  • CryptoLibrary.dll
  • ETicketLibrary.dll
  • FileSystemMetaLibrary.dll
  • Lz4Library.dll
  • YamlDotNet.dll
  • YamlDotNet.Dynamic.dll

使用方法

AuthoringTool の各機能の利用方法を説明します。


警告：

nca, nsp の内容表示、展開、エントリ修正の機能は、SDK パッケージに同梱されている nca, nsp に対して実行することができません。

警告：

リソースデータのルートディレクトリに "!" から始まる名前のファイルやディレクトリを含めようとするとエラーになります。これは今後の更新で修正される可能性があります。

nca ファイルを作成する

createnca サブコマンドを利用します。

 $ AuthoringTool.exe help createnca
createnca: Create new nintendo content archive from .adf or specified directory and .nmeta and .desc file.
Usage: AuthoringTool.exe createnca [-o <outputPath>] <inputAdfPath>
Usage: AuthoringTool.exe createnca [-o <outputPath>] [--save-adf] [--only-adf] [--filter <fdfPath>] --meta <metaFilePath> --desc <descFilePath> --program <codePath> [<dataPath>]

引数一覧

位置引数はありません。

オプション一覧

参考：

"AuthoringTool.exe createnca [-o <outputPath>] <inputAdfPath>" のコマンド形式では、createnca コマンドで作成した adf ファイルから nca ファイルを作成することができます。
この際 -o 以外のオプションは無視されます。adf ファイルのフォーマットは予告なく変更されますので、ご利用の際はご注意ください。

-o

出力先のパスを指定します。ディレクトリを指定した場合は、そのディレクトリ以下に output.nca というファイル名で出力されます。

パスが指定するディレクトリが存在しない場合は、自動的に作成します。指定しなかった場合は、実行パスの直下に "output.nca" というファイル名で出力されます。
--save-adf nca ファイル作成時の中間ファイル（.adf）を削除せずに残します。nca ファイルに含まれているファイルの内容などが確認できます。現在、中間ファイルを利用する機能は提供されていませんが、将来、エントリレイアウト編集機能を提供する予定です。
--only-adf nca ファイル作成時の中間ファイル（.adf）を生成のみ行い、nca ファイルを生成せずに処理を終了します。短い処理時間で中間ファイルの内容を確認するのに利用します。
--filter

リソースデータファイルをフィルタアウトするためのフィルタ記述ファイルを指定します。詳細は後述の「フィルタ記述ファイル（Filter Description File）を記述する」を参照してください。
このオプションを指定しなかった場合は、dataPath で指定したディレクトリ以下のすべてのディレクトリ及びファイルが格納されます。

フィルタ記述ファイルは nmeta ファイルの中で指定することも可能です。詳細は「nmeta ファイルの書き方」を参照してください。
--meta .nmeta ファイルのパスを指定します。このオプションは必須です。
--desc

.desc ファイルのパスを指定します。--program オプションを利用する場合、このオプションは必須です。

.desc ファイルは提出時の確認に使用する場合がありますので、パッケージのファイルを個別にコピーやリネームせず、パッケージから直接参照するようにしてください。
--program nca をプログラムとして作成します。codePath および dataPath にはディレクトリを指定します。

指定されたディレクトリ内のコードデータ及びリソースデータを格納します。リソースデータを含めない場合、dataPath は省略することができます。
参考：

拡張子が .meta から .nmeta に変更されましたが、 今まで通り .meta ファイルもご利用いただけます。


使用例

  • NX-VSI で出力されたコードデータ、リソースデータから nca ファイルを作成する
$ AuthoringTool.exe createnca -o sample.nca --meta $NINTENDO_SDK_ROOT/Resources/SpecFiles/Application.aarch64.lp64.nmeta --desc $NINTENDO_SDK_ROOT/Resources/SpecFiles/Application.desc --program C:/Application/Binaries/NX64/NX_Debug/application.nspd/program0.ncd/code C:/Application/Binaries/NX64/NX_Debug/application.nspd/program0.ncd/data


nsp ファイルを作成する

creatensp サブコマンドを利用します。

 $ AuthoringTool.exe help creatensp
creatensp: Create new Nintendo Submission Package from .adf or specified directory and .nmeta and .desc file.
Usage: AuthoringTool.exe creatensp [-o <outputPath>] <inputAdfPath>
Usage: AuthoringTool.exe creatensp [-o <outputPath>] [--save-adf] [--only-adf] [--filter <fdfPath>] --meta <metaFilePath> --type <contentMetaType> [--desc <descFilePath> --program <codePath> [<dataPath>] [--nro <nroDirectoryPath>] --icon <language1> <iconPath1> <language2> <iconPath2> ... --nx-icon <language1> <iconPath1> <language2> <iconPath2> ...] [--error-unpublishable [--xml]]  [--no-check-dir-warning]
注意：

リソースデータにおいてファイルを持たないディレクトリ（空のディレクトリやフィルタアウトすることで空になるディレクトリ）は nsp ファイルには含められません。

引数一覧

位置引数はありません。

オプション一覧

参考：

"AuthoringTool.exe creatensp [-o <outputPath>] <inputAdfPath>" のコマンド形式では、creatensp コマンドで作成した adf ファイルから nsp ファイルを作成することができます。
この際 -o 以外のオプションは無視されます。adf ファイルのフォーマットは予告なく変更されますので、ご利用の際はご注意ください。

-o

出力先のパスを指定します。ディレクトリを指定した場合は、そのディレクトリ以下に output.nsp というファイル名で出力されます。

パスが指定するディレクトリが存在しない場合は、自動的に作成します。指定しなかった場合は、実行パスの直下に "output.nsp" というファイル名で出力されます。
--save-adf nsp ファイル作成時の中間ファイル（.adf）を削除せずに残します。nsp ファイルに含まれているファイルの内容などが確認できます。現在、中間ファイルを利用する機能は提供されていませんが、将来、エントリレイアウト編集機能を提供する予定です。
--only-adf nsp ファイル作成時の中間ファイル（.adf）を生成のみ行い、nsp ファイルを生成せずに処理を終了します。短い処理時間で中間ファイルの内容を確認するのに利用します。
--filter

リソースデータファイルをフィルタアウトするためのフィルタ記述ファイルを指定します。詳細は後述の「フィルタ記述ファイル（Filter Description File）を記述する」を参照してください。
このオプションを指定しなかった場合は、dataPath で指定したディレクトリ以下のすべてのディレクトリ及びファイルが格納されます。

フィルタ記述ファイルは nmeta ファイルの中で指定することも可能です。詳細は「nmeta ファイルの書き方」を参照してください。
--meta .nmeta ファイルのパスを指定します。このオプションは必須です。
--desc

.desc ファイルのパスを指定します。--program オプションを利用する場合、このオプションは必須です。

.desc ファイルは提出時の確認に使用する場合がありますので、パッケージのファイルを個別にコピーやリネームせず、パッケージから直接参照するようにしてください。
--type アプリケーションの種別を指定します。現在は Application, SystemProgram が指定可能です。通常のアプリケーション作成時は Application を指定します。
--program プログラムを指定します。codePath および dataPath にディレクトリを指定した場合、プログラムとしての nca を作成し格納します。プログラムにリソースデータを含めない場合、dataPath は省略することができます。
codePath にファイルを指定した場合、そのファイルをプログラムとしての nca ファイルと解釈しそのまま格納し、dataPath に指定した内容は無視されます。
この際、指定する nca ファイルは --meta、--desc で与えたのと同じファイルを指定して作成したものでなければなりません。
--html-document

オフライン HTML のディレクトリを指定します。

参考：

NX Addon 3.6.0 以降で廃止予定、もしくは NX Addon 4.x では廃止されたオプションです。代わりに、 nmeta ファイルの書き方 の HtmlDocumentPath をご利用ください。
--accessible-urls

アプリケーションからどのサイトに接続できるかの情報が書かれたドキュメントのディレクトリを指定します。

参考：

NX Addon 3.6.0 以降で廃止予定、もしくは NX Addon 4.x では廃止されたオプションです。代わりに、 nmeta ファイルの書き方 の AccessibleUrlsFilePath をご利用ください。
--legal-information

「ソフトリーガル情報」用のドキュメントのZIP ファイルを指定します。

参考：

NX Addon 3.6.0 以降で廃止予定、もしくは NX Addon 4.x では廃止されたオプションです。代わりに、 nmeta ファイルの書き方 の LegalInformationFilePath をご利用ください。
--nro

.nro ファイルが含まれるのディレクトリを指定します。

詳細はこちらを参照して下さい。
--icon

アイコン画像を指定します。

詳細はこちらを参照して下さい。

参考：

NX Addon 3.6.0 以降で廃止予定、もしくは NX Addon 4.x では廃止されたオプションです。代わりに、 nmeta ファイルの書き方 の IconPath をご利用ください。
--nx-icon

--icon で生成される画像に問題がある場合に使用します。

詳細はこちらを参照して下さい。

参考：

NX Addon 3.6.0 以降で廃止予定、もしくは NX Addon 4.x では廃止されたオプションです。代わりに、 nmeta ファイルの書き方 の NxIconPath をご利用ください。
--error-unpublishable

nsp ファイルが任天堂に提出する条件に満たない状態だった場合にエラーにします。

提出する条件については、 「マスターROM提出手順書」の「マスターROMのアップロード」に書かれている「ROMチェックでのエラー一覧」を参照してください。
--no-check-dir-warning

リソースデータにおいてファイルを持たないディレクトリ（空のディレクトリやフィルタアウトすることで空になるディレクトリ）が存在する場合に表示される警告メッセージを抑制します。

参考：

NX Addon 6.0.0 以降で利用できます。
--xml --error-unpublishable の結果を xml 形式で出力します。
参考：

拡張子が .meta から .nmeta に変更されましたが、 今まで通り .meta ファイルもご利用いただけます。

使用例

  • NX-VSI で出力されたコードデータ、リソースデータから nsp ファイルを作成する
 $ AuthoringTool.exe creatensp -o sample.nsp --meta $NINTENDO_SDK_ROOT/Resources/SpecFiles/Application.aarch64.lp64.nmeta --type Application --desc $NINTENDO_SDK_ROOT/Resources/SpecFiles/Application.desc --program C:/Application/Binaries/NX64/NX_Debug/application.nspd/program0.ncd/code C:/Application/Binaries/NX64/NX_Debug/application.nspd/program0.ncd/data --nro C:/Application/Binaries/NX64/NX_Debug/application.nspd/program0.ncd/data/nro


パッチ nsp ファイルを作成する

  • パッチのビルド、実行方法 の makepatch コマンドを参照してください。

Usage: AuthoringTool.exe list <inputPath>
Options:

  --original <originalPath>            Original application nsp path. This is required if <inputNspPath> is a patch.

引数一覧

<inputNspOrNcaPath>
 入力の対象となる nsp/nca ファイルのパスを指定します。

オプション一覧

--targetregex 特定のパスのエントリのみを正規表現で絞り込んで表示する場合に指定します。正規表現については、.NET Framework の記法に対応しています。
--original パッチの適用先となる元アプリケーション nsp ファイルのパスを指定します。パッチ nsp ファイルの内容を表示する際に必要です。
--patched-only パッチに含まれるファイルのみを表示する際に使用します。対象ファイルがパッチ nsp ファイルでない場合には影響は与えません。
--patched-only-in-detail

パッチに含まれるファイルのみを断片化情報とともに表示する際に使用します。また、元となる nsp から削除されたファイルも別途表示します。

 対象ファイルがパッチ nsp ファイルでない場合には影響は与えません。

断片化情報の見方については後述します。

    • 元のアプリケーションから削除されたファイルがある場合 "Listing removed files from original..." と表示された後にファイル名とサイズが表示されます。

元アプリケーションとパッチとの差分情報は list コマンドの --patched-only オプションをご利用ください。

使用例

$ AuthoringTool.exe diffpatch sample.nsp sample_v1_patch.nsp sample_v2_patch.nsp
! 157933e46d19226339f7da0e8f7b8118.nca/fs0/main         (114859 byte)
+ 157933e46d19226339f7da0e8f7b8118.nca/fs1/data01.bin   (3145784 byte)
- 157933e46d19226339f7da0e8f7b8118.nca/fs1/data02.bin   (19802 byte)
! 157933e46d19226339f7da0e8f7b8118.nca/fs1/data03.bin   (64797 -> 65581 byte)
! f570f0fd8b748ef3f40542fed287e52a.cnmt.nca/fs0/Patch_010000000000b9c9.cnmt     (356 -> 612 byte)
    • 行頭が '!' のファイルは、比較元パッチと比較先パッチで内容が異なるファイルを表します
      • ファイルサイズが変わっている場合、変更前、変更後のサイズをそれぞれ表示しています。
    • 行頭が '+' のファイルは、比較元パッチに存在せず、比較先パッチで追加されたファイルを表します
    • 行頭が '-' のファイルは、比較元パッチに存在し、比較先パッチで削除されたファイルを表します
    • 比較元パッチと比較先パッチで内容が同じファイルは表示されません。

フィルタ記述ファイル（Filter Description File）を記述する

フィルタ記述ファイルはリソースデータに含めるファイルを制限する目的で使用する、ユーザ記述のテキスト形式ファイルです。
フィルタ記述ファイルの拡張子は .fdf とすることを推奨します。

参考：

記述したフィルタが意図通りに動作しているかを確認するのに --only-adf オプションを利用することができます。

仕様

フィルタ記述ファイルでは次のフィルタリングルールを記述することができます。

  • 除外ルール
  • 例外ルール
  • フィルタリング対象エントリ指定ルール

各ルールの対象として記述するエントリ（ディレクトリまたはファイル）のパスは、リソースデータディレクトリからの相対パスで表現します。
また、正規表現については、.NET Framework の記法に対応します。

除外ルール

正規表現にマッチするエントリをリソースデータとして含めないようにします。

例外ルール

正規表現にマッチするエントリを、除外ルールの対象から外します。
除外ルールと例外ルールが同じエントリに対してそれぞれ複数回適用される場合、最後に適用されたルールが優先されます。

フィルタリング対象エントリ指定ルール

指定したエントリをリソースデータに含めます。本ルールでは正規表現を使用することができません。
このルールが適用された場合、このルールで指定されたエントリ以外は以降の除外ルール・例外ルールの対象にはならず、リソースデータにも含められません。
このルールは複数回記述することができますが、このルールの前に除外ルールおよび例外ルールを記述することはできません。

ディレクトリパスを指定した場合は、そのディレクトリ以下のファイルが全てこのルールの対象となります。

参考：

リソースデータディレクトリ以下にある多数のエントリから部分的に多数のエントリを選択しリソースデータに含める場合など、独立する除外ルールおよび例外ルールの記述量と対象エントリ数が大きいときにはフィルタリング処理に時間がかかることがあります。
このとき代わりにフィルタリング対象エントリ指定ルールを利用してリソースデータを選択することで、より短時間でフィルタリング処理を行うことができる場合があります。

処理

--icon オプションについて

--icon オプションは nsp に含めるアイコン画像を指定するオプションです。

参考：

NX Addon 3.6.0 以降で廃止予定、もしくは NX Addon 4.x では廃止されたオプションです。代わりに、 nmeta ファイルの書き方 の IconPath をご利用ください。

入力する画像は下記の仕様を満たす必要があります。

  • 画像サイズが 1024 * 1024
  • 形式が 24Bit BMP


入力された画像からは下記の仕様の 2 種類のベースライン形式の Jpeg が生成され nsp に格納されます。

  • 大きなサイズの Jpeg
    • NX 以外のデバイスで使用される可能性があります。
    • サイズは 1024 * 1024


  • 小さなサイズの Jpeg 
    • NX のホーム画面等のアイコンとして表示されます。
    • サイズは 256 * 256（入力された画像を縮小します。）
    • ファイルサイズが 100KB を超えない範囲で最高品質となるように調整されます。

--nx-icon オプションについて

--nx-icon オプションは --icon オプションで生成される小さなサイズの Jpeg を直接指定するオプションです。

参考：

NX Addon 3.6.0 以降で廃止予定、もしくは NX Addon 4.x では廃止されたオプションです。代わりに、 nmeta ファイルの書き方 の NxIconPath をご利用ください。

AuthoringTool の生成する小さなサイズの Jpeg に問題がある場合でのみ使用を検討してください。

このオプションで小さなサイズの Jpeg を指定する場合、その見え方が大きなサイズのものと同等になるようにしてください。
--icon で指定した画像と根本的なデザインが異なるものを指定してはいけません。 


入力する画像は下記の仕様を満たす必要があります。

  • サイズが 256 * 256
  • ベースライン形式の Jpeg
  • ファイルサイズが 100KiB 以下

入力する画像の Exif は独自のものに置き換えられた状態で nsp に格納されます。
