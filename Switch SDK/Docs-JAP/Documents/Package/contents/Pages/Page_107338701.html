リソースデータ（ROM） (NX Addon)

 

NX アプリケーションでは、Windows プラットフォームでビルドしたアプリケーションと同様に nn::fs::MountRom() でリソースデータを参照できますが、リソースデータの配置場所が異なります。

同じリソースデータに対して異なる名前を指定して重複して参照することができます。NX アプリケーションでは同時に 10 回まで重複して参照することができます。

NX プラットフォームでビルドしたアプリケーション実行時のリソースデータの配置について

アプリケーションのリソースデータの配置場所はアプリケーションの実行形式によって異なります。
プロジェクトプロパティ＞構成プロパティ＞全般の Application Program Format で Nsp 形式、Raw 形式を選択できます。この違いについては NX アプリケーションを開発する を参照してください。
また、アプリケーションの開発機へのインストールについては NX アプリケーションを開発機にインストールして実行する を参照してください。

Nsp 形式でビルドして実行した場合

nsp ファイルのデータ領域に格納されたファイルをリソースデータとして参照します。

NX アプリケーションプロジェクトでソリューションプラットフォームを NX プラットフォーム用のものに変更し、プロジェクトプロパティの「オーサリング＞全般＞アプリケーションデータディレクトリ」にディレクトリパスを指定することで、指定したディレクトリ以下のファイルがビルドで出力される nsp ファイルのデータ領域に含まれます。

リソースデータを取り込んだ nsp ファイルの作成はビルド中のリンク処理の後に行われるため、事前にリソースデータの変換処理などを行う場合は、リンク処理よりも前に済ませておく必要があります。

アプリケーションを開発機にインストールして実行した場合

nsp ファイル内のデータ領域に格納されたファイルをリソースデータとして参照します。
AuthoringTool で nsp ファイルを作成する際に、データ領域に含めるリソースデータを指定できます。

Raw 形式でビルドして実行した場合

ビルド成果物の出力先の nspd ディレクトリに含まれる "data" ディレクトリ以下のファイルをリソースデータとして参照します。

NX アプリケーションプロジェクトでソリューションプラットフォームを NX プラットフォーム用のものに変更し、プロジェクトプロパティの「オーサリング＞全般＞アプリケーションデータディレクトリ」 にディレクトリパスを指定することで、指定したディレクトリのジャンクションを data ディレクトリとしてビルド時に作成します。

参考：

ジャンクションはネットワークドライブ上のディレクトリに対して作成することはできません。そのため、"アプリケーションデータディレクトリ" にネットワーク上に配置したリソースデータを指定するとビルドエラーになります。
そのようなリソースデータを利用したい場合は、一度ジャンクションが作成可能なドライブにそのリソースデータへのシンボリックリンクを作成し、"アプリケーションデータディレクトリ" にそのシンボリックリンクを指定することで対応できます。
ただし、シンボリックリンクの作成には Windows の管理者権限が必要です。
また、ジャンクションは NTFS 以外でフォーマットされたボリューム上に作成することはできません。ビルド成果物の出力ディレクトリは NTFS でフォーマットされたボリューム上に設定するようにしてください。

アプリケーション実行中のリソースデータの更新について

Nsp 形式のアプリケーション実行中にリソースデータを更新することはできません。アプリケーションイメージの更新とアプリケーションの再起動が必要です。

Raw 形式のアプリケーション実行中にリソースデータの更新を行うには、事前に nn::fs::Unmount() を実行するなどして PC 上のリソースデータがどのアプリケーションプロセスからも参照されていない状態にする必要があります。
nn::fs::MountRom() によって参照が有効になっている PC 上のリソースデータを更新した場合、その更新がアプリケーションに正しく反映されることは保証されていません。

また、いずれの実行形式においても、nn::fs::MountHost() と nn::fs::MountHostRoot() では事前に nn::fs::Unmount() を実行せずとも PC 上のファイルやディレクトリの更新が反映されることが保証されます。
