パッチ適用による File System パフォーマンスの変化

パッチの適用前後では、File System の読み込み速度が変化します。パッチ適用後はパッチと元ストレージの両方の情報を参照してファイル読み込みを行うことになるため、パッチで修正を行ったファイルだけでなく未修正のファイルも含めて読み込み性能は下がります。

性能の低下は以下の二つによって起こります。

  • パッチ適用後のシーケンシャルアクセス・ランダムアクセス速度の変化
  • ファイル実体がパッチ適用後に断片化されることによる変化

それぞれについて説明します。

パッチ適用後のシーケンシャルアクセス・ランダムアクセス速度の変化

パッチ適用後は、適用前に比べてこれらのアクセス速度が低下します。Firmware for NX 1 系ではファームの不具合により想定以上に遅くなっているため、特に注意してください。

なお、パフォーマンスはシステムの負荷、および今後のファームウェアの修正によっても変動するので、あくまでも参考値とします。

注意：

コードのみの修正の場合、Firmware for NX 1 系に存在する不具合を踏まない可能性があります。一度目のパッチ提出では速度の問題が無かったとしても、二度目以降に問題が無いかは必ず確認するようにしてください。

ファイルの実体が断片化されることによる変化

NX のパッチはファイルの一部に対しての修正が可能となっているため、修正内容やマッチングの結果によってはファイルの実体が断片化されます。

例えば、offset: 0 から size: 64KiB は元アプリを参照、offset: 64KiB から 2KiB はパッチ側を参照、offset: 66KiB から 14KiB はパッチの別の場所を参照というような場合です。

この場合、各々の領域をランダムアクセスでアクセスした場合の速度になると考えます。具体的な速度は、初版アプリケーションおよびパッチのインストールされているストレージと断片化の仕方によるので、断片化が特に激しい場合は注意して確認を行ってください。

断片化の状況は、AuthoringTool の list コマンドで --patched-only-in-detail オプションを付けることで確認できます。例えば先ほどの例だと下記のように表示されます。

00112233445566778899aabbccddeeff.nca/fs1/file/binary.dat
[Original]
(offset, size) = (0, 65536)
[Patch]
(offset, size) = (65536, 2048), (67584, 14336)
