Compiler

リンカに gold を使用すると、ld でリンク時に検出されるエラーが実行時まで検出されない場合がある

-fuse-ld=gold.exe オプションを指定することで gold リンカを使うようにした場合（VSI では「gold リンカーを使用」を有効にした場合）、ld ではリンク時に検出されるエラーが実行時まで検出されない場合があります。

NVN

ディスプレイがフレームバッファを更新するタイミング以外のタイミングでもディスプレイリリースフェンスがシグナルする

症状

nvnWindowAcquireTexture() が返す NVNsync オブジェクト（ディスプレイリリースフェンス）は通常、ディスプレイコントローラーがスキャンするフレームバッファが VSync 時に切り替わるタイミングでシグナルしますが、次のタイミングでもシグナルする場合があります。

  • フォーカス状態がインフォーカス状態と他の状態の間で遷移するタイミング
  • ドックイン・ドックアウトや HDMI ケーブル挿抜によりディスプレイが切り替わるタイミング
  • スリープにする・スリープを解除するタイミング
  • 本体設定でテレビ出力の解像度を変更したタイミング

これらの場合、ディスプレイリリースフェンスがシグナルしても nvnQueuePresentTexture() で発行された GPU コマンドが完了したことが保証されません。アプリケーションの CPU 処理が GPU 処理の完了をディスプレイリリースフェンスによって判定していると、上記タイミングで GPU が使用中のコマンドバッファやその他のバッファを破壊、もしくは書き換えてしまう可能性があります。

回避策

GPU 処理の完了待ちは、ディスプレイリリースフェンスによって間接的に待つのではなく、nvnQueueFinish() もしくは nvnQueueFenceSync() と nvnSyncWait() によって直接待つようにしてください。

修正予定

修正可否を検討中です。修正時期は未定です。

特定の条件でカラーバッファと深度バッファが物理メモリを共有する場合にレンダリング結果にノイズが生じることがある

症状

カラーバッファと深度バッファで物理メモリプールを共有している際、以下の条件がすべて当てはまる場合、レンダリング結果にノイズが生じる可能性があります。

  • 物理メモリプールにマップされる仮想メモリプールのうち、少なくともひとつが 16-bit depth (Z16) のストレージクラスである
  • 物理メモリプールにマップされる仮想メモリプールのうち、少なくともひとつが compressible color のストレージクラスである
  • 物理メモリプールにマップされる仮想メモリプールのうち、少なくともひとつが NVN_MEMORY_POOL_FLAGS_GPU_CACHED_BIT で作成されている
回避策

上記の条件にすべて当てはまるように同じ物理メモリプールに仮想メモリプールをエイリアスしないようにすることで、この問題を回避できます。
または、上記のようにエイリアスされているカラーテクスチャをクリアする際、fast clear として登録されていない値を使うことでも回避することができます。

修正予定

現在問題を調査中です。修正時期は未定です。

GLSLC 関連

  • EOL markers must be present in #included files or else a compilation error will occur.
  • The TX1 versions of the GLSLC library may produce slightly different machine code than the Windows version, including potential performance differences. The code produced on both platforms should be functionally correct though, and convergence of the two will happen in future changes.
  • FP16 can be accomplished in GLSL using mediump/lowp precision qualifiers with the extension __GL_NV_desktop_mediump_lowp, or by using explicit float16 types with the NV_gpu_shader5 extension.  When using the extension GL_NV_desktop_mediump_lowp with mediump and lowp qualifiers, there are a few known issues:
    • Mixing interface attribute types between shader stages with explicit NV_gpu_shader5 float16 types on one side of the interface and mediump/lowp types on the other side of the interface could produce linker errors.
    • Using explicit NV_gpu_shader5 float16 types as the types for interface attributes between two shader stages will produce a linker error.
    • ints/uints can not be implicitly converted to floats qualified with mediump/lowp qualifiers.  This can be worked around by explicit constructing float types from the int/uint variables (similar to explicit casting in C/C++).  This issue currently affects GLSL shaders using GL_NV_desktop_mediump_lowp _as well as_ ES shaders regardless of whether the extension GL_EXT_shader_implicit_conversions is used.
    • FP16 math when used with imageStore may cause corruption in certain cases. This issue is being investigated and will be fixed in an upcoming release.

WEB

 Webアプレットのアクセス可能なURLの記述のしかたによって、メモリ不足が起こりエラーが発生したりWebアプレットが終了してしまう

症状

Webアプレットで、アクセス可能なURLの正規表現に ".*" などの長い文字列にマッチする表現を使用し、かつ、ページ遷移時の判定の際に長い URL と判定が行われると、メモリ不足が起こりエラーが発生したりWebアプレットが終了してしまう場合があります。

回避策

アクセス可能なURLの判定は「完全一致」ではなく「部分一致」で行われるので、正規表現を必要十分な文字列にマッチするように置き換えると回避できる場合があります。特にWebコンテンツの開発中は、".*" や "^https?://.*$" のように記述して全ての URL とマッチさせたい場合がありますが、"^https?://" という記述に置き換えることができます。

 Webアプレット・オフラインWebアプレットの動画専用起動で、起動時の画面の種類が反映されない。

症状

Webアプレット・オフラインWebアプレットで動画専用起動を指定した時、起動時の画面の種類を指定していても反映されません。

回避策

回避策はありません。

不具合が修正された後に挙動が変わってしまうため、不具合が修正されるまで、動画専用起動を指定した時は起動時の画面の種類を指定しないでください。
