概要

本ページでは、任天堂で想定しているパッチの開発イテレーションについて説明します。

必ずしもこのイテレーションに従う必要はありませんので、参考としてください。

修正版アプリケーションのビルドとデバッグ

まずはパッチを意識せずに、修正版のアプリケーションをビルド・デバッグして開発を行います。修正版のアプリケーションの作成なので、VSI などでそのまま開発できます。

修正版アプリケーションと初版アプリ＋パッチとでは、ストレージアクセス速度以外は同じと考えることができます。そのため、速度が影響するところ以外は基本的に修正版アプリケーションで動作確認を行うことができます。

初版アプリケーションからのセーブデータの互換性確認も、ここで行うことが可能です。その確認のために初版アプリケーションと修正版アプリケーションを行き来するときには、必須バージョンによって起動が制限される場合があります。制限された場合は、DevMenuCommand で必須バージョンのリセットを行ってください。また行き来を行うと、修正版アプリケーションで作成したセーブデータが、初版アプリケーションから読み込まれることになります。それに不都合がある場合は、セーブデータも併せて削除してください。

パッチのビルドとデバッグ

速度の影響や、最終提出物の QA 確認は、パッチをビルドしてデバッグする必要があります。また、パッチのサイズには上限を設けるので、そこに収まるかの確認のためにもパッチをビルドする必要があります。具体的なサイズはパッチガイドを参照してください。

パッチのビルド、および作成したパッチを使った動作確認の方法は、パッチのビルド、実行方法 を参照してください。読み込み速度に影響するので、直前のパッチが存在する場合は必ず直前のパッチを指定してください。

現時点では、VSI からパッチ適用済みの状態で起動することはできません。一方、パッチのビルド、実行方法 の方法で実行を開始した後に、あとから VSI でアタッチしてデバッグすることは可能です。あとからアタッチする方法は、ビルド済プログラムをデバッグする の「既に起動しているアプリをデバッグする」を参照してください。

オンカードパッチ付き NX カードの製造を予定している場合は、オンカードパッチ付き NX 開発カードでの動作確認も必ず行ってください。
